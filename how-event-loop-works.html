Node.js Event Loop

A visual guide to how Node.js handles asynchronous operations

The 6 phases — executed in order on every cycle
Phase 1
Timers
Runs callbacks scheduled by setTimeout() and setInterval() whose delay has expired. The delay is a minimum wait, not a guarantee — if the loop is busy, the callback fires late.
setTimeout(fn, 0) → fires here
Phase 2
Pending callbacks
Executes I/O callbacks that were deliberately deferred by the previous cycle. The most common case is TCP-level errors such as ECONNREFUSED. Rarely touched in application code.
TCP ECONNREFUSED → deferred here
Phase 3
Idle / Prepare
Internal use only. Node.js uses this phase to prepare for the upcoming Poll phase. Not accessible from userland JavaScript — you will never schedule anything here directly.
— internal to Node.js —
Phase 4
Poll ★
The most important phase. Retrieves new I/O events and executes their callbacks — HTTP requests, DB results, file reads, network responses. If the queue is empty and no timers are pending, the loop blocks here waiting for new events.
HTTP, DB, fetch, stream data → all here
Phase 5
Check
Runs callbacks registered with setImmediate(). Always executes right after Poll in the same cycle — this makes it reliably earlier than setTimeout(fn, 0) when called inside an I/O handler.
setImmediate(fn) → fires here
Phase 6
Close callbacks
Handles close events for sockets and streams that were destroyed abruptly. Runs last so all pending I/O for that resource has already been processed. This is where stream cleanup lands.
socket / stream 'close' event → fires here
Priority queues — run between every phase (and between every callback)
process.nextTick()
Highest priority of all. The entire nextTick queue is drained after every single callback, before any Promise microtask and before the loop advances to the next phase. Used by Node.js internals (EventEmitter, streams) to defer state transitions safely.
process.nextTick(fn) → before any .then()
Promise microtasks
Runs after nextTick, but still before the loop moves on. The entire queue is drained — not just one item. If a .then() enqueues another .then(), that runs too. A deep Promise chain can therefore starve the I/O queue.
Promise.resolve().then(fn) · async/await
One thread, non-blocking. JavaScript runs on a single thread — but I/O (disk, network, DB) is delegated to libuv, which uses a thread pool and the OS async I/O APIs. When the work is done, the callback re-enters the loop through the Poll phase. The main thread is free the entire time.
Worked example
Five concurrent requests — what happens inside the loop
GET
/users
Two parallel requests. Each queries the database via db.query(), which is delegated to the libuv thread pool. Both DB results arrive in the same tick and are processed as I/O callbacks in Poll.
libuv thread pool
GET
/externalUsers
Two parallel requests. Each calls an external API via fetch(). When the OS signals a response, the Promise resolves and the .then() lands in the microtask queue — not the I/O queue.
OS network I/O → microtask
POST
/uploadReport
Pipes the request body as a Readable stream directly into an S3 upload. Each file chunk fires a data event in Poll. When the upload completes, the stream is destroyed and its close event fires in phase 6.
stream → Poll · close → phase 6
The handler for GET /users (A) also registers a setTimeout(fn, 0) and a setImmediate(fn) to demonstrate the ordering difference between the two. All five requests arrive simultaneously and all async results are assumed to land in the same tick for clarity.

The key insight about multiple microtasks ⑨ and ⑩: After DB callback A ⑧ completes, Node.js drains the entire microtask queue before picking the next I/O callback. Both fetch .then() A and fetch .then() B are in that queue, so they both run — one after the other — before DB callback B ⑪ gets a turn. It is not "one microtask per I/O callback" — it is "all pending microtasks, in order, before the next I/O callback".

Backpressure: if S3 is slower than the incoming data, write() returns false → the Readable pauses automatically → resumes when drain fires in Poll. No blocking, no busy-waiting — all event-driven.

stream close ⑭ is the only stream event that does not land in Poll — it goes to phase 6 Close callbacks, guaranteeing that all other processing on that stream has already completed before cleanup runs. This is why phase 6 exists.

Rule 1 — Full drain
The microtask queue is fully emptied after every single callback — not just one item. All pending .then() and nextTick() calls run before the loop moves on.
Rule 2 — Recursive risk
If a microtask queues another microtask, that one runs too — still before the next I/O callback. A chain of .then() calls can starve the I/O queue indefinitely.
Rule 3 — nextTick first
process.nextTick() is drained before Promise microtasks. Within each queue, callbacks run in the order they were queued — FIFO.
